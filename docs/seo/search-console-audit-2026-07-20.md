# Search Console SEO audit — DocGPT.ai

**Дата аудита:** 2026-07-20

**Источник данных:** SearchConsole.ai, подключённый Google Search Console

**Статус:** redacted; локальные исправления выполнены, деплой не выполнялся

> Этот файл не содержит идентификаторов Search Console, токенов, данных доступа, полного экспорта запросов или иных закрытых строк. Числа приведены только в агрегированном виде, достаточном для принятия решений.

## Краткий вывод

Главная SEO-возможность сейчас — не выпускать ещё больше страниц, а повысить качество и связность уже опубликованного массива. В данных обнаружены **929 URL, впервые появившихся в доступной выборке GSC**, из которых **846 не получили ни одного клика**. Это сильный сигнал, что рост количества страниц опережает их способность получать поисковый спрос.

Самые быстрые и контролируемые действия:

1. остановить массовый выпуск новых SEO-страниц до проверки качества;
2. исправить 20 страниц, где структура контент-плана попала в title, description, FAQ и видимый текст как `H2:`;
3. улучшить страницы с уже понятным коммерческим намерением;
4. добавить внутренние ссылки с релевантных страниц;
5. удалить неподтверждённые рейтинги, отзывы и обещания продукта;
6. после деплоя измерять результат не раньше, чем Google успеет переобойти страницы.

## Как применён playbook SearchConsole.ai

Анализ выполнен по принципам SEO analysis playbook:

- сначала проверен общий тренд и масштаб набора URL;
- отдельно рассмотрены страницы с показами без кликов и страницы с коммерческим намерением;
- возможности приоритизированы по потенциальному влиянию, уверенности и трудозатратам;
- изменения разделены на быстрые on-page правки, внутреннюю перелинковку, технический контроль качества и последующие эксперименты;
- выводы не основаны только на средней позиции: учитывались вероятность индексации, соответствие намерению, CTR и качество посадочной страницы;
- для программного контента применён контроль повторяемости, фактической полезности и шаблонных дефектов.

## Правила принятия решения по 929 URL

Не удалять страницы только потому, что у них пока нет кликов. Для каждого кластера проверить:

1. возраст URL и был ли у Google достаточный период на переобход;
2. индексируется ли страница и совпадает ли canonical;
3. есть ли показы и по каким типам намерения;
4. отличается ли страница от соседних URL фактической полезностью;
5. есть ли внутренняя ссылка из тематического hub;
6. соответствует ли страница реальной функции продукта;
7. можно ли объединить её с более сильной страницей без потери intent;
8. есть ли бизнес-ценность у запроса, даже если объём мал.

Результат review должен присваивать один из статусов: `keep`, `improve`, `merge`, `redirect`, `noindex` или `remove`. Для destructive-решений нужен отдельный утверждённый список URL.

## Ограничения и необходимые решения

- Изменения находятся только в локальном рабочем дереве; публикация на Netlify не выполнялась.
- Нужна проверка владельца workflow, существует ли внешний Hermes scheduler и остановлен ли он.
- Дата повторного измерения должна отсчитываться от фактического deploy, а не от даты локального изменения.
- Для claims о пользователях, рейтинге, моделях, тарифах и возможностях конкурентов нужен проверяемый первичный источник и дата проверки.
- Перед merge/redirect/noindex для массового cohort нужен отдельный URL-level review.

## Redaction note

В отчёте сохранены агрегаты и публичные URL-пути. Не сохранены сырые запросы, идентификаторы property/account, OAuth-данные, токены, персональные данные и полный экспорт Search Console.
